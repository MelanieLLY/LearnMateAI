# Homework 5: LLM Coding Aids - Final Retrospective

## Part 1: How did the custom skill change our workflow? What tasks became easier?
Introducing the `/add-feature` custom skill transformed our workflow, turning an unpredictable process into a standardized pipeline. Initially, when building the first feature (Student Notes), we had to stop and think between each phase. We often guessed at test cases during the RED phase and created files randomly during the GREEN phase, which led to "circular import errors" that had to be debugged.

Based on these specific pain points, we iterated the skill to **v2**, teaching a **7-phase process** for building features (`EXPLORE → PLAN → RED → GREEN → REFACTOR → COMMIT → DOCUMENT`). When we applied v2 to the second feature (Flashcards), the results were highly reliable. The new skill enforced a **4-part checklist** during the EXPLORE phase; it locked in a **6-scenario test template** for the RED phase (1 happy path, plus 5 errors like 422 missing/empty field, 401 not logged in, 403 wrong user role, and 404 missing resource); most importantly, it strictly enforced the file creation order (`Model → Schema → Service → Router → Dependencies → Main`).

The most direct change was that less thinking equaled faster work. By following the pattern, we passed all tests on the first try with zero import errors, and maintained clean, atomic git commits with clear messages. The skill made us perfectly achieve a fundamental mindset shift: it made us go **"from thinking about HOW to build features to focusing on WHAT features to build."**

## Part 2: What did MCP integration enable that wasn't possible before?
Integrating the Model Context Protocol (MCP) successfully broke down the information barrier between our local terminal environment and the cloud-based project management platform. Before connecting the GitHub MCP, if we wanted the large model to assist in programming, we had to leave the terminal, manually check the Sprint board in our code repository via a browser, or manually run the `gh issue list` command and copy the text over to Claude. The AI's working context had serious "blind spots."

While configuring the MCP, we deeply experienced the conflict between "AI hallucinations" and network verification protocols. Initially, the assistant defaulted to a configuration plan aimed at enterprise users (`claude mcp add --transport http...`), which triggered an `Incompatible auth server (does not support dynamic client registration)` error. After digging through the course materials and the official `install-claude.md` source code documentation, we discovered that standard users must explicitly inject authentication info. We eventually corrected our configuration: we applied the standard `add-json` method and manually attached an HTTP PAT Header with strict formatting (paying attention to spaces) like `"Authorization":"Bearer YOUR_TOKEN"`. *([MCP Connection Verification Screenshot: docs/screenshot/10_mcp_connected.png](screenshot/10_mcp_connected.png))*

Once fully connected, the MCP empowerment was huge. Claude Code gained the permission to read and intervene in our entire project directly within the command-line interface. For example, when we issued a command, it could instantly invoke the `Queried github` tool and accurately grab the latest active requests in the repository: #9 (Instructor Module Management Dashboard), #8 (Production & Polish Architecture), and #7 (System Eval & MLOps). Even more impressively, it also read the Labels and specific text details inside the Issues. Now, without humans acting as messengers, the large model can actively grab real requirements from the cloud and generate matching code, truly achieving a high degree of synchronization between the requirements context and the development context. *([Issues Fetching Terminal Output Screenshot: docs/screenshot/11_mcp_fetch_issues.png](screenshot/11_mcp_fetch_issues.png))*

## Part 3: What would we build next (Hooks/Agents)?
In future development, we hope to further deepen the large model's complex toolchain integration capabilities. Currently, our MCP application is mostly limited to simple one-way queries (like checking Issues). Since we've proven that Claude Code can perfectly understand the project architecture, we plan to explore a more cohesive automated version control and ticket management pipeline (Collaboration Pipeline).

Instead of introducing highly complex external testing pipelines, we hope to build a stable, one-stop workflow right within the current GitHub system. The specific vision is: when a Feature is marked as developed and passes all tests, we can command Claude to execute this combination of commands:
1. Extract the recently changed code, automatically generate a Conventional Commits standard Git Commit message, and commit it.
2. Using the GitHub MCP, automatically draft a detailed Pull Request (PR) description based on the submitted content.
3. Automatically update the label status for the corresponding source Issue in the main branch (e.g., from `in-progress` to `in-review`).
4. Finally, assist in completing the code Merge and closing that Issue along with it.

In this envisioned closed loop, we upgrade the AI's role from a mere "code generator" to a "project process secretary." The entire pipeline will strictly adopt a Human-in-the-loop confirmation mechanism—meaning every step must wait for the developer to manually confirm with a `Y` before proceeding to the next step. This will not only dramatically reduce the operations and maintenance costs of our code during the Project 3 sprint, but it will also allow the AI to maximize its value in a fixed workflow safely and reliably.
